## 1. Architecture

### Overall Shape

The project follows a conventional layered Spring Boot monolith:

```
Controller → Service → Repository → MySQL (prod) / H2 (test)
```

The frontend is a feature-slice Redux Toolkit SPA with Axios and React Router v6.

**What works well:**
- Feature-slice Redux organisation mirrors the backend's domain boundaries cleanly.
- `ApiResponse<T>` wrapper is consistent across all endpoints.
- `GlobalExceptionHandler` centralises error mapping.
- `OpenApiConfig` + Springdoc 2.3 gives auto-generated Swagger UI at `/swagger-ui.html`.

### ARCH-01 — No service interfaces (except `LeaveService`)

Only `LeaveService` follows the `Service` / `ServiceImpl` split. All other services (`TimesheetService`, `AuthService`, `EmployeeService`, `ReportService`) are concrete `@Service` classes injected directly. This tightly couples controllers to implementations and makes unit testing harder (can't mock via an interface without Mockito spies).

**Fix:** Extract interfaces for all services, or at minimum document that the concrete-class pattern is intentional.

---

### ARCH-02 — `ReportService` inner DTOs

```java
public class ReportService {
    @Data @AllArgsConstructor
    public static class DepartmentUtilization { ... }
    @Data @AllArgsConstructor
    public static class EmployeeEffort { ... }
}
```

Inner static DTO classes inside a `@Service` are not idiomatic Spring and can't be independently tested or reused by other services.

**Fix:** Move all report DTOs to `com.timekeeper.dto.response.report`.

---

### ARCH-03 — `TimesheetService` is a 350+ line god class

`TimesheetService` handles: creation, week validation, entry CRUD, overlap detection, holiday/leave conflict checks, duration calculation, AND response mapping. It has no interface and no collaborator injection.

**Fix (incremental):**
1. Extract `TimesheetMapper` (entity → DTO conversion).
2. Extract `TimesheetValidationService` (overlap, holiday, leave conflict checks).
3. Introduce `TimesheetService` interface + `TimesheetServiceImpl`.

## 2. Security

### SEC-01 (P0) — Gmail App Password exposed in `application-dev.properties`

```properties
# application-dev.properties line 21
spring.mail.password=cuuxclfejlpnvpgi
```

This file is tracked in git. Anyone with repository access can send emails as the configured Gmail account.

**Fix:**
```properties
spring.mail.password=${MAIL_PASSWORD}
```
Rotate the App Password immediately. Add `application-dev.properties` to `.gitignore` and provide `application-dev.properties.example` with placeholder values.

---

### SEC-02 (P0) — `GET /timesheets/{timesheetId}` has no ownership check

**File:** `TimesheetController.java`

```java
@GetMapping("/{timesheetId}")
@PreAuthorize("hasAnyRole('EMPLOYEE', 'MANAGER', 'ADMIN')")
public ResponseEntity<ApiResponse<TimesheetResponse>> getById(
        @PathVariable Long timesheetId, @AuthenticationPrincipal Employee currentUser) {
    return ResponseEntity.ok(ApiResponse.success(..., timesheetService.getById(timesheetId)));
}
```

The `currentUser` parameter is received but never used to scope the query. Any authenticated employee can read any other employee's timesheet by incrementing the ID.

**Fix:**
```java
TimesheetResponse ts = timesheetService.getById(timesheetId);
if (currentUser.getRole() == Employee.Role.EMPLOYEE
        && !ts.getEmployeeId().equals(currentUser.getId())) {
    throw new AccessDeniedException("Access denied");
}
```

---

### SEC-03 (P0) — Manager can approve any employee's leave — not just direct reports

**File:** `LeaveServiceImpl.java` — `approveLeave()`

The method confirms the approver's ID and the leave's `approvedBy` field are set, but never checks that the leave belongs to one of the manager's direct reports.

**Fix:**
```java
if (!leave.getEmployee().getManagerId().equals(approverId)) {
    throw new AccessDeniedException("You can only approve leaves for your direct reports");
}
```

---

### SEC-04 — MANAGER can see any employee's timesheet report (not just their team)

**File:** `ReportController.java` — `GET /reports/employee-timesheet`

```java
@GetMapping("/employee-timesheet")
@PreAuthorize("hasAnyRole('MANAGER', 'ADMIN')")
public ResponseEntity<...> getEmployeeTimesheetReport(@RequestParam String employeeId, ...)
```

No check that `employeeId` is a direct report of the calling manager.

**Fix (service layer):**
```java
if (callerRole == Role.MANAGER) {
    Employee target = employeeRepository.findById(employeeId).orElseThrow(...);
    if (!managerId.equals(target.getManagerId())) {
        throw new AccessDeniedException("Employee is not in your team");
    }
}
```

---

### SEC-05 — `ChangePasswordRequest` has no password constraints

**File:** `ChangePasswordRequest.java`

```java
@NotBlank
private String newPassword;   // "a" is a valid password
```

**Fix:**
```java
@NotBlank
@Size(min = 8, message = "Password must be at least 8 characters")
private String newPassword;
```

---

### SEC-06 — `spring.jpa.hibernate.ddl-auto=update` in base `application.properties`

`update` silently migrates the schema in any environment that picks up the base properties. DDL should be managed explicitly via Flyway/Liquibase.

**Fix:** Change to `validate` in base properties. Keep `create-drop` in `application-test.properties`.

## 3. Performance

### PERF-01 (P1) — `getAll()` in `EmployeeService` is an unbounded full table scan

```java
public List<EmployeeResponse> getAll() {
    return employeeRepository.findAll()  // zero pagination
            .stream()
            .map(this::toResponse)
            .toList();
}
```

Every call to `/employees` loads every employee record into the JVM heap.

**Fix:** Add `Pageable` parameter:
```java
public Page<EmployeeResponse> getAll(Pageable pageable) {
    return employeeRepository.findAll(pageable).map(this::toResponse);
}
```

---

### PERF-02 — `getDepartmentUtilization` has O(N×M) DB queries

```java
for (Department dept : departments) {                           // N depts
    for (Employee emp : employees) {                            // M employees
        timesheetRepository.findByEmployeeIdAndWeekStartDate(…) // 1 query
        timeEntryRepository.sumHoursLoggedByTimesheetId(…)      // 1 query
    }
}
```

5 departments × 20 employees = 100+ queries per page load.

**Fix:** One aggregate JPQL query grouping by department with a `SUM` projection.

---

### PERF-03 — `LeaveServiceImpl.toResponse()` fires one DB query per leave (N+1)

```java
approvedByName = employeeRepository.findById(leave.getApprovedBy())…  // N+1
```

`getTeamLeaves()` with 20 employees × 10 leaves = 200 extra queries.

**Fix:** Use a `LEFT JOIN FETCH` in the repository query or project the approver name via a `@Query`.

---

### PERF-04 — `getProjectEffort` loads all time entries into memory for aggregation

```java
List<TimeEntry> entries = timeEntryRepository.findByProjectId(projectId);
// then: entries.stream().mapToDouble(…).sum()
```

**Fix:** `@Query("SELECT SUM(te.hoursLogged) FROM TimeEntry te WHERE te.project.id = :projectId GROUP BY te.employee.id")`

---

### PERF-05 — Entry add/update/delete always makes 2 sequential HTTP calls

```js
await timesheetService.addEntry(timesheetId, data)            // mutate
const response = await timesheetService.getById(timesheetId)  // full re-fetch
```

The backend can return the updated `TimesheetResponse` from the entry endpoint directly, eliminating the second round trip on every entry operation.

---

### PERF-06 — No `@Transactional(readOnly = true)` on read methods

All service read methods (`getById`, `getAll`, `getTeamLeaves`, etc.) participate in a read-write transaction unnecessarily. `readOnly = true` tells Hibernate to skip dirty checking and enables potential routing to a read replica.

**Fix:** Annotate all read-only service methods:
```java
@Transactional(readOnly = true)
public TimesheetResponse getById(Long id) { … }
```

## 4. Scalability

### SCALE-01 — `TokenBlacklist` is an in-memory `ConcurrentHashMap`

The blacklist lives in the application process. With multiple instances (load balancer / horizontal scale), a token blacklisted on instance A is still valid on instance B.

**Fix:** Replace with a Redis `SET` with TTL matching token expiry, or store JTI in a DB table with an index on `jti + expiredAt`.

---

### SCALE-02 — `LoginRateLimiter` is in-process per-IP state

Same issue as TokenBlacklist — per-IP attempt counters are not shared across instances.

**Fix:** Redis `INCR` + `EXPIRE` on a key like `login:attempt:{ip}` gives cluster-safe rate limiting.

---

### SCALE-03 — No pagination on any list endpoint

`getAllTimesheets()`, `getTeamLeaves()`, `findByEmployeeIdOrderByWeekStartDateDesc()`, `getAllEmployees()` — all return unbounded result sets. As data grows these become full table scans.

---

### SCALE-04 — Email sending is synchronous on the request thread

```java
emailService.sendTimesheetSubmittedEmail(…);  // called inline in TimesheetService
```

If Gmail is slow or unavailable, the user's "Submit timesheet" request hangs.

**Fix:** `@Async` on `EmailService` methods or push to a queue (SQS / RabbitMQ).

## 5. Code Quality

### QUAL-01 — `ordinal()` used to compute entry day date — fragile

**File:** `TimesheetService.java`

```java
LocalDate dayDate = timesheet.getWeekStartDate().plusDays(request.getDay().ordinal());
```

Works today because `MONDAY=0`. If the enum is ever reordered, this silently produces wrong dates.

**Fix:**
```java
private static final Map<TimeEntry.DayOfWeek, Long> DAY_OFFSET = Map.of(
    TimeEntry.DayOfWeek.MONDAY, 0L,
    TimeEntry.DayOfWeek.TUESDAY, 1L,
    TimeEntry.DayOfWeek.WEDNESDAY, 2L,
    TimeEntry.DayOfWeek.THURSDAY, 3L,
    TimeEntry.DayOfWeek.FRIDAY, 4L
);
LocalDate dayDate = timesheet.getWeekStartDate().plusDays(DAY_OFFSET.get(request.getDay()));
```

---

### QUAL-02 — `AuthService.login()` queries the DB twice for the same user

```java
authenticationManager.authenticate(…);                              // DB call 1 via UserDetailsService
Employee employee = employeeRepository.findByEmail(request.getEmail()); // DB call 2
```

**Fix:**
```java
Authentication auth = authenticationManager.authenticate(…);
Employee employee = (Employee) auth.getPrincipal();
```

---

### QUAL-03 — `@Valid` missing on update request DTOs in controllers

`UpdateEmployeeRequest` and `UpdateProjectRequest` are received without `@Valid` so all constraint annotations on those DTOs are silently ignored.

**Fix:** Add `@Valid` to every `@RequestBody` parameter where the DTO has constraints.

---

### QUAL-04 — `DataInitializer` hardcodes year 2026

```java
.startDate(LocalDate.of(2026, 1, 1))
.endDate(LocalDate.of(2026, 12, 31))
```

Will produce stale seed data in 2027+. Use `LocalDate.now().withDayOfYear(1)` relative dates.

---

### QUAL-05 — `GlobalExceptionHandler` leaks internal messages

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<…> handleGeneral(Exception ex) {
    return ResponseEntity.status(500)
            .body(ApiResponse.error("An unexpected error occurred: " + ex.getMessage()));
}
```

`ex.getMessage()` on unhandled exceptions can expose SQL queries, class names, and stack traces.

**Fix:**
```java
log.error("Unhandled exception", ex);
return ResponseEntity.status(500)
        .body(ApiResponse.error("An unexpected error occurred. Please contact support."));
```

---

### QUAL-06 — `DataIntegrityViolationException` unhandled — 500 with raw SQL error

Duplicate email registration currently returns HTTP 500 with a raw SQL constraint message.

**Fix:**
```java
@ExceptionHandler(DataIntegrityViolationException.class)
public ResponseEntity<…> handleDataIntegrity(DataIntegrityViolationException ex) {
    return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(ApiResponse.error("A record with this value already exists"));
}
```

## 6. UX & Frontend State

### UX-01 (P1) — `fetchMyTimesheets` and `fetchAllTimesheets` write to the same Redux array

**File:** `timesheetSlice.js`

Both thunks set `state.timesheets = action.payload`. When an ADMIN navigates from "All Timesheets" to "My Timesheets", the store silently overwrites the previous list. The opposite trip causes the same collision in reverse.

**Fix:** Use two separate slice fields: `state.myTimesheets` and `state.allTimesheets`.

---

### UX-02 — Manager approve/reject only updates `myLeaves`, not `teamLeaves`

**File:** `leaveSlice.js`

```js
.addCase(approveLeave.fulfilled, (state, action) => {
    state.myLeaves = state.myLeaves.map(…)
    // teamLeaves NOT updated → stale PENDING badge until refresh
})
```

**Fix:**
```js
.addCase(approveLeave.fulfilled, (state, action) => {
    state.myLeaves   = state.myLeaves.map(l => l.id === action.payload.id ? action.payload : l)
    state.teamLeaves = state.teamLeaves.map(l => l.id === action.payload.id ? action.payload : l)
})
// Same fix for rejectLeave.fulfilled
```

---

### UX-03 — `PROJECT_COLORS` has no overflow guard in `TimesheetDetail.jsx`

```js
const PROJECT_COLORS = ['#6366F1', '#10B981', '#F59E0B', '#EF4444', '#8B5CF6', '#EC4899', '#14B8A6', '#F97316']
// project 9+ → PROJECT_COLORS[index] === undefined → timeline bar renders with no color
```

**Fix:** `PROJECT_COLORS[index % PROJECT_COLORS.length]`

---

### UX-04 — Leave application blocks past dates

```java
if (request.getStartDate().isBefore(today)) {
    throw new BusinessException("Cannot apply leave for a past date");
}
```

Real employees routinely log sick leave retroactively. This is a critical SaaS workflow gap.

---

### UX-05 — Notification bell shows hardcoded unread badge permanently

**File:** `Layout.jsx`

```js
const SAMPLE_NOTIFICATIONS = [{ id: 1, …, unread: true }, …]
```

Always renders the badge. Users will learn to ignore it. Label as "Coming Soon" or remove.

---

### UX-06 — No 404 page — unknown routes silently redirect to `/`

```jsx
<Route path="*" element={<Navigate to="/" replace />} />
```

Users mistyping URLs get bounced with no explanation.

## 8. Data Model Gaps

### DATA-01 (P1) — `Timesheet.TimesheetStatus` only has `DRAFT` and `SUBMITTED`

```java
public enum TimesheetStatus {
    DRAFT, SUBMITTED   // ← APPROVED and REJECTED are missing
}
```

The frontend `STATUS_CFG` map and all badge/color logic references `APPROVED` and `REJECTED` as real states. The backend has no approval workflow — timesheets can never reach `APPROVED`. This is a fundamental data contract mismatch between the two layers.

**Fix:**
1. Add `APPROVED`, `REJECTED` to the enum.
2. Add `approvedBy` (FK to Employee) and `rejectionReason` fields to `Timesheet`.
3. Add `POST /timesheets/{id}/approve` and `POST /timesheets/{id}/reject` endpoints (MANAGER/ADMIN only).
4. Add ownership check: only the submitter's manager or an ADMIN can approve.

---

### DATA-02 — No audit fields on `Timesheet` or `Leave`

Neither entity tracks `createdAt`/`updatedAt`. Useful for reporting ("submitted on time vs. late") and debugging.

**Fix:** Add a `@MappedSuperclass AuditEntity` with JPA `@CreatedDate`/`@LastModifiedDate`.
