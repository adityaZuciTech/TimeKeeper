### CRIT-01 — Credentials hardcoded in `application.properties`

**File:** `backend/src/main/resources/application.properties`

```properties
spring.datasource.username=root
spring.datasource.password=root
spring.mail.password=nigmnpcqxnotrwcr   # ← real Gmail App Password exposed
jwt.secret=404E635266556A586E3272357538782F413F4428472B4B6250645367566B5970
```

This file is committed to GitHub. The Gmail App Password and JWT secret are **live credentials in a public repository**. Anyone with repo access can send emails as your Gmail account and forge valid JWTs for any user indefinitely.

**Fix:**
```properties
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
spring.mail.password=${MAIL_PASSWORD}
jwt.secret=${JWT_SECRET}
```
Add `application.properties` to `.gitignore`, use a `application.properties.example` template. Rotate the Gmail App Password immediately.

---

### CRIT-02 — JWT logout does nothing server-side

**File:** `AuthController.java`

```java
@PostMapping("/logout")
public ResponseEntity<ApiResponse<Void>> logout() {
    // JWT is stateless; client discards token
    return ResponseEntity.ok(ApiResponse.success("Logged out successfully", null));
}
```

A stolen or intercepted token remains valid for the full 24 hours after the user "logs out." There is no token blacklisting.

**Fix (minimum viable):** Implement a token blacklist table (`revoked_tokens`) or Redis set. On logout, persist the token's JTI (JWT ID claim). In `JwtAuthenticationFilter`, reject tokens whose JTI is in the blacklist.

---

### CRIT-03 — Silent exception swallowing in `JwtAuthenticationFilter`

**File:** `JwtAuthenticationFilter.java`

```java
} catch (Exception e) {
    // Invalid token - continue without authentication
}
```

Catches every possible exception silently — including nulls and DB errors. No log trail for security events.

**Fix:**
```java
} catch (ExpiredJwtException e) {
    log.debug("JWT expired for request: {}", request.getRequestURI());
} catch (JwtException e) {
    log.warn("Invalid JWT token: {}", e.getMessage());
} catch (Exception e) {
    log.error("Unexpected error in JWT filter", e);
}
```

### CRIT-04 — `GlobalExceptionHandler` leaks internal error messages

**File:** `GlobalExceptionHandler.java`

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<ApiResponse<Void>> handleGeneral(Exception ex) {
    return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(ApiResponse.error("An unexpected error occurred: " + ex.getMessage()));
}
```

`ex.getMessage()` on unhandled exceptions can expose stack traces, SQL queries, class names, and file paths to clients.

**Fix:**
```java
log.error("Unhandled exception", ex);
return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
        .body(ApiResponse.error("An unexpected error occurred. Please contact support."));
```

## 2. Bugs Found

### BUG-01 — `ordinal()` used to derive day date in `addEntry` — fragile

**File:** `TimesheetService.java`

```java
LocalDate dayDate = timesheet.getWeekStartDate().plusDays(request.getDay().ordinal());
```

Works today because enum order is `MONDAY=0, TUESDAY=1...`. If the enum ever changes order, this silently computes the wrong date with no test catching it.

**Fix:** Use an explicit offset map:
```java
private static final Map<TimeEntry.DayOfWeek, Long> DAY_OFFSET = Map.of(
    TimeEntry.DayOfWeek.MONDAY, 0L, TimeEntry.DayOfWeek.TUESDAY, 1L,
    TimeEntry.DayOfWeek.WEDNESDAY, 2L, TimeEntry.DayOfWeek.THURSDAY, 3L,
    TimeEntry.DayOfWeek.FRIDAY, 4L
);
LocalDate dayDate = timesheet.getWeekStartDate().plusDays(DAY_OFFSET.get(request.getDay()));
```

### BUG-02 — Admin/Manager receives wrong status code when trying to submit another user's timesheet

**File:** `TimesheetController.java` + `TimesheetService.java`

The service correctly throws `BusinessException("You can only submit your own timesheets")`, but `BusinessException` maps to **400 Bad Request** in `GlobalExceptionHandler`. The correct response for an authorization failure is **403 Forbidden**.

**Fix:** Create a separate `ForbiddenException extends RuntimeException` mapped to 403, or throw `AccessDeniedException` (already handled as 403).

---

### BUG-03 — MANAGER can pull any employee's timesheet report, not just their team

**File:** `ReportController.java`

```java
@GetMapping("/employee-timesheet")
@PreAuthorize("hasAnyRole('MANAGER', 'ADMIN')")
public ResponseEntity<...> getEmployeeTimesheetReport(
        @RequestParam String employeeId, ...) {
```

No check that `employeeId` belongs to the calling manager's team. A manager can view timesheets for employees in any department.

**Fix in service layer:**
```java
if (callerRole == Role.MANAGER) {
    Employee target = employeeRepository.findById(employeeId)...;
    if (!managerId.equals(target.getManagerId())) {
        throw new AccessDeniedException("Employee is not in your team");
    }
}
```

### BUG-04 — `AddTimeEntryRequest` has no validation on WORK-required fields

**File:** `AddTimeEntryRequest.java`

```java
private String projectId;    // no @NotNull
private LocalTime startTime; // no @NotNull
private LocalTime endTime;   // no @NotNull
```

A direct API call with `entryType=WORK` but null `startTime` causes a `NullPointerException` in `Duration.between(null, null)`.

**Fix:** Add a cross-field validator or `@AssertTrue` method:
```java
@AssertTrue(message = "projectId, startTime, and endTime are required for WORK entries")
public boolean isWorkEntryValid() {
    if (entryType != TimeEntry.EntryType.WORK) return true;
    return projectId != null && startTime != null && endTime != null && startTime.isBefore(endTime);
}
```

### BUG-05 — `ChangePasswordRequest` / `CreateEmployeeRequest` have no minimum password length

**File:** `ChangePasswordRequest.java`, `CreateEmployeeRequest.java`

```java
@NotBlank
private String newPassword;  // "a" is valid
```

**Fix:**
```java
@NotBlank
@Size(min = 8, message = "Password must be at least 8 characters")
private String newPassword;
```

### BUG-06 — Timesheet with zero entries can be submitted

**File:** `TimesheetService.java`

`submit()` does not check whether the timesheet has any entries. An employee submits an empty timesheet → reminder system stops sending them emails → utilization reports show 0 hours for a "submitted" week.

**Fix:**
```java
List<TimeEntry> entries = timeEntryRepository.findByTimesheetId(timesheetId);
if (entries.isEmpty()) {
    throw new BusinessException("Cannot submit a timesheet with no entries");
}
```

### BUG-07 — `leaveSlice` `approveLeave`/`rejectLeave` only update `myLeaves`, not `teamLeaves`

**File:** `frontend/src/features/leaves/leaveSlice.js`

```js
.addCase(approveLeave.fulfilled, (state, action) => {
    state.myLeaves = state.myLeaves.map(l => l.id === action.payload.id ? action.payload : l)
    // teamLeaves NOT updated → stale status in manager's view
})
```

Manager approves a leave from the team list but sees it still as PENDING until page refresh.

**Fix:**
```js
.addCase(approveLeave.fulfilled, (state, action) => {
    state.myLeaves   = state.myLeaves.map(l => l.id === action.payload.id ? action.payload : l)
    state.teamLeaves = state.teamLeaves.map(l => l.id === action.payload.id ? action.payload : l)
})
```
Same fix for `rejectLeave.fulfilled`.

### BUG-08 — `PROJECT_COLORS` array in `TimesheetDetail.jsx` has no overflow guard

**File:** `TimesheetDetail.jsx`

```js
const PROJECT_COLORS = ['#6366F1', '#10B981', '#F59E0B', '#EF4444', '#8B5CF6', '#EC4899', '#14B8A6', '#F97316']
```

If a user has more than 8 projects, `PROJECT_COLORS[index]` returns `undefined` → timeline bar renders with no color.

## 4. Code Smells

---

### SMELL-01 — Report DTOs are inner static classes of the `@Service`

```java
public class ReportService {
    @Data @AllArgsConstructor
    public static class DepartmentUtilization { ... }
```

Inner classes in a `@Service` are not idiomatic Spring. Makes the service file oversized and the DTOs impossible to reuse independently.

**Fix:** Move all report response DTOs to `com.timekeeper.dto.response.report` package.

---

### SMELL-02 — `TimesheetService` is a 350+ line god class

Handles: creation, submission, entry management, overlap validation, holiday/leave blocking, AND response mapping — all in one class with no interface.

**Fix:** Extract `TimesheetMapper`, `TimesheetValidationService`, introduce `TimesheetService` interface + `TimesheetServiceImpl`.

---

### SMELL-03 — `AuthService.login()` queries the DB twice for the same user

```java
authenticationManager.authenticate(...); // → DB call via UserDetailsService
Employee employee = employeeRepository.findByEmail(request.getEmail()) // → second DB call
```

**Fix:**
```java
Authentication auth = authenticationManager.authenticate(...);
Employee employee = (Employee) auth.getPrincipal();
```

---

### SMELL-04 — `DataInitializer` hardcodes 2026 dates

```java
.startDate(LocalDate.of(2026, 1, 1))
.endDate(LocalDate.of(2026, 12, 31))
```

Will become stale in 2027. Use `LocalDate.now()` relative dates.

---

### SMELL-05 — `@Valid` annotation missing on update request DTOs in controllers

`UpdateEmployeeRequest` and `UpdateProjectRequest` are received in controllers without `@Valid`, so any constraint annotations on those DTOs are never evaluated.

## 5. Security Issues

---

### SEC-01 — Real credentials committed to git (see CRIT-01)

Gmail App Password + JWT secret in `application.properties` pushed to GitHub.

---

### SEC-02 — No rate limiting on `POST /auth/login`

Unlimited brute-force password attempts possible. No lockout, no CAPTCHA, no attempt counter.

**Fix:** Add Bucket4j + a `LoginAttemptService` that tracks IP-based failed attempts and locks after N failures.

---

### SEC-03 — MANAGER can access any employee's data via reports API (see BUG-03)

Missing ownership check on `GET /reports/employee-timesheet`.

---

### SEC-04 — `allowedHeaders("*")` with `allowCredentials(true)` in CORS config

```java
config.setAllowedHeaders(List.of("*"));
config.setAllowCredentials(true);
```

Overly permissive. Restrict to: `["Authorization", "Content-Type", "Accept"]`.

---

### SEC-05 — JWT has no token invalidation (see CRIT-02)

No blacklist → stolen tokens are valid until expiry.

---

### SEC-06 — JWT secret is minimum-length (256-bit)

The hardcoded secret decodes to exactly 32 bytes — the minimum HMAC-SHA256 key size. No entropy buffer. Use a randomly generated 512-bit secret.

## 6. Performance Issues

---

### PERF-01 — `getDepartmentUtilization` has O(N×M) DB queries

```java
for (Department dept : departments) {              // N depts
    for (Employee emp : employees) {               // M employees
        timesheetRepository.findByEmployeeIdAndWeekStartDate(...)  // 1 query each
        timeEntryRepository.sumHoursLoggedByTimesheetId(...)       // 1 query each
    }
}
```

5 departments × 20 employees = **100+ queries per Organization page load.**

**Fix:** Single aggregate JPQL query grouping by department.

---

### PERF-02 — `getProjectEffort` loads all time entries into memory

```java
List<TimeEntry> entries = timeEntryRepository.findByProjectId(projectId);
```

A multi-year project = thousands of rows loaded into a Java list just to sum them.

**Fix:** Use `SUM(te.hoursLogged) GROUP BY emp.id` in the repository query.

---

### PERF-03 — `LeaveServiceImpl.toResponse()` fires 1 DB query per leave to resolve approver name

```java
approvedByName = employeeRepository.findById(leave.getApprovedBy())...  // N+1
```

`getTeamLeaves()` with 20 employees × 10 leaves each = 200 extra queries.

---

### PERF-04 — Entry add/update/delete always makes 2 sequential HTTP calls

```js
await timesheetService.addEntry(timesheetId, data)         // call 1
const response = await timesheetService.getById(timesheetId)  // call 2
```

Every mutation makes a full re-fetch. Backend should return the updated `TimesheetResponse` directly from entry endpoints.

---

### PERF-05 — No pagination on any list endpoint

`getAllTimesheets()`, `getTeamLeaves()`, `findByEmployeeIdOrderByWeekStartDateDesc()` — all return unbounded result sets. Will become full table scans over time.

## 7. Suggested Improvements

---

### Add password strength validation

```java
@Pattern(
  regexp = "^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).{8,}$",
  message = "Password must be 8+ chars with uppercase, lowercase and digit"
)
private String newPassword;
```

---

### Add `@Transactional(readOnly = true)` on all read-only service methods

Enables Hibernate optimizations: no dirty checking, no flush on commit, potential routing to read replicas.

---

### Handle `DataIntegrityViolationException` in `GlobalExceptionHandler`

Duplicate email registration currently returns a 500 with a raw SQL constraint message. Map to 409 Conflict:

```java
@ExceptionHandler(DataIntegrityViolationException.class)
public ResponseEntity<ApiResponse<Void>> handleDataIntegrity(DataIntegrityViolationException ex) {
    return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(ApiResponse.error("A record with this value already exists"));
}
```

---

### Use `application-dev.properties` / `application-prod.properties` profiles

Separate dev (show-sql, debug logging) from prod (validate DDL, info logging) using Spring profiles.
